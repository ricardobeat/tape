## High Impact ŌĆö RegExp (libregexp swap)

- [ ] **B32** ŌĆö Unicode regex correctness: regex compilation now honors `\u{1F600}`, `\p{Letter}`, and `v`-flag string-sets, but `lre_exec` is invoked with `cbuf_type=0` (byte mode). That works for ASCII-only subjects and for property-escapes against a pure-ASCII input, but multi-byte UTF-8 subject strings break: `"\u{1F600}".match(/\u{1F600}/u)` returns `null` because libregexp interprets `cbuf_type=0` as "one byte per character" and never decodes the surrogate pair. Fix: convert UTF-8 subject bytes to UTF-16 in our wrapper and pass `cbuf_type=2` (matching QuickJS's wiring in `quickjs.c:47884`). When this lands the test262 `regexp-unicode-property-escapes` count (currently all-CE or fail) will start passing through, and `regexp-v-flag` closer to full conformance.

- [ ] **B33** ŌĆö `d`-flag `match.indices` runtime: we now accept `d` at compile time (and call out `hasIndices=true` correctly), but `String.prototype.match` and `RegExp.prototype.exec` never populate `.indices` on the result array. Also `RegExp.prototype.hasIndices` accessor getter is not registered (the test262 skip list does not currently skip it, so it shows as fail rather than CE). Fix: extend the regex call sites to (a) extract capture offsets from the exec result and build an `indices` array of `[start,end]` pairs (and `indices.groups`), and (b) add `BUILTIN_REGEXP_PROTO_GET_HASINDICES` + `REGEXP_PROTO_GET_UNICODESETS` accessors mirroring the existing `dotAll`/`sticky` pattern in `src/builtins/regexp.c3` (~line 1265).

- [ ] **B34** ŌĆö `String.prototype.match` with a non-global named-group regexp: `String.prototype.match(/(?<y>\d{4})/)` returns the plain array of the full match, but `result.groups` is left `undefined`. `RegExp.prototype.exec(/(?<y>\d{4})/.exec("2024-01"))` correctly returns `result.groups === {y:"2024"}`. Root cause in `src/builtins/string.c3` ~line 2110 (`builtin_string_proto_match` symbol-match fast path): after `re_exec` it sets the full match but never reads `re_get_groupnames(re)` to construct the `groups` object. Fix: mirror the same groupname iteration that `regexp_proto_exec` (`src/builtins/regexp.c3` ~line 354) already does, populating `result.groups[name] = capture_string` for each non-empty named capture. (Once B33 lands, populate `result.indices.groups` the same way.)

- [ ] **B37** ŌĆö `destructuring-binding` feature flag fires on 2,233 tests that produce unexpected CEs. Largest single contributor to "parser too strict" CE bucket. Sampled cases show the parser rejects ES2015 destructuring patterns that the spec allows in positions our parser doesn't yet cover ŌĆö confirmed by `duktape_c3` error on `built-ins/.../target-assign-throws-iterator-return-get-throws.js` and on `statements/try/scope-catch-param-lex-open.js` (`compile error at line 16:10: expected '<identifier>', got '['` ŌĆö likely catch-clause destructuring pattern not supported). Notable distinct surfaces to investigate: (a) `catch ([x, _]) {}` ŌĆö catch-clause destructuring, (b) `for ([x, y] of arr) {}` ŌĆö for-of destructuring head, (c) `[a = function() { return b; }, b = 1]` ŌĆö default-value-side-effect-ordering interaction, (d) `({ [computed()]: a } = obj)` ŌĆö destructuring with computed property name, (e) `({ __proto__: x, ...y } = obj)` ŌĆö destructuring rest with computed proto. Each surface may be a separate fix in the parser/desugar path. Repro for the catch case: `test/test_b37_destructuring_catch.js` (draft). Requires ~200-phrase investigation per surface to identify exact parser hole.

- [x] **B38** ŌĆö *(done ŌĆö root cause was B43, now fixed; see B43 entry)* `generators` feature flag CEs. The bulk was missing generator-method shorthand in class bodies/object literals, resolved by the generators branch merge. Residual generator-adjacent CEs (computed-property-names from `yield`/`await` expressions) are tracked under B44's cluster, not a separate surface.

- [ ] **B39** ŌĆö `default-parameters` feature flag fires on 1,250 tests producing unexpected CEs. We claim `default-parameters` is supported. Cases likely split between (a) destructured default values (`function f({ a = 1 }) {}` we may be misparsing as parameter with default), (b) default values in arrow-function shorthand (`(a = 1) => a`), (c) default values with TDZ references (`function f(a = b, b = 1) {}`), (d) computed `[[]]= expr` patterns, (e) default-parameter side-effect ordering tests. Likely shares root cause with B37 (destructuring + default value combined). Same investigation pattern. Cheaper than B37/B38 because the surface area is narrower. *(Partial progress in `3559c1f` + `143c534`: outer-defaults on destructured params and multi-param destructured defaults now parse.)*

## Session 250 review findings (2026-07-05) ŌĆö see plans/040-test262-100-percent.md for full detail

- [x] **B40** ŌĆö *(done, session 251)* Huge-length array-like handling. Root cause: `array_to_length`/`array_to_length_checked` clamped to 2^53-1 correctly but then truncated to 32-bit `uint`, so end-iterating methods started at index ~4.29e9 instead of ~9e15 and walked billions of indices interning a numeric string each one (the 30GB MEMKILL mechanism). Fixed in `src/builtins/array.c3`: lengths/indices widened to `ulong` across all Array.prototype methods; `MAX_ARRAY_LENGTH` const + `tval_set_index` (fastint is 48-bit ŌĆö larger indices/lengths stored as doubles); pre-mutation TypeError in `push`/`unshift`/`splice`/`concat` when result would exceed 2^53-1; `find`/`findIndex`/`findLast`/`findLastIndex` no longer skip holes (spec Gets every index ŌĆö the hole-skip was the residual infinite loop); `pop`/`slice`/`reverse`/`fill`/`copyWithin` moved from ToUint32 to ToLength with spec hole/delete semantics; `splice` no-arg now deletes 0 (was: everything) and always writes length. Cluster result: 24/28 of `*/maximum-index` + `*-integer-limit*` pass; the 4 remaining need `Proxy` (not implemented).
