The `if (a >= 65 && a <= 90) { … } else if (a >= 97 && a <= 122) { … } else { … }` pattern compiles to `CMP + IF_FALSE + LE + LDREG + IF_FALSE_bridge + body`, and the peephole's `fused_off = sbx + 1` made the synthesized JMP_Nxx target the bridge IF_FALSE (the one that bridges to the next branch). When the JMP_Nxx fired (c not in 65-90), it landed on the bridge — but the bridge's `rA` was the *previous* iteration's `c <= 90` result (the JMP_Nxx skipped the `LE` that would have refreshed it), so the bridge mis-fires and the wrong body runs. Fix: in the peephole, detect when the if_false's target is itself an if_false/if_true (the "and-continuation bridge" pattern) and extend the fused offset to jump past the bridge too: `fused_off = sbx + 1 + bridge_sbx + 1`. This also closes the standalone "all-uppercase" repro `test/test_b26_l3.js` (where else was a constant string) — same root cause, different surface. Rosette went from 5 failures (B23/B24/B26 trio) to 2 (B25 number parser, B28 array mutation) and `bench-fast` is unchanged.**

The `JMP_Nxx` does NOT materialize the comparison result in `left` (the destination register the `&&` operator uses as its final result). The fix was correctly handling the if/else-if/else case from B23/B26 — but it fired indiscriminately for any if_false/if_true at the destination, including the if_true *back edge* of a `while (cond1 && cond2)` loop. The structural difference: in the B23/B26 if-chain, the destination is a FORWARD continuation (`if_false` with positive offset, jumping to the next chain link). In a `while` loop with `&&`, the if_false short-circuit for cond1 lands on the LDREG that precedes the if_true back edge, and the if_true is a BACK edge with NEGATIVE offset. Extending `fused_off` by `bridge_sbx` (= negative offset) flipped the fused `JMP_Nxx` from a forward short-circuit to a backward loop-forever jump. Once the loop counter underflowed past the condition threshold, the fused JMP_Nxx kept firing and the loop spun (or, for heapsort, the bridge-skipping made the array appear to grow).

A negative or zero offset means it's a loop back edge or self-targeting branch — leave `bridge_sbx = 0` and let the fused JMP_Nxx land on the bridge slot unchanged. This preserves the B23/B26 fix (forward continuations still skip correctly) while letting the while-loop back edge be the actual target.

- [x] **B30** — `stack.js` `while`-loop variant (from `test/rosetta/FAILURES.md`): "`while (!st.isEmpty()) out += st.pop()` returns a partial/garbled string when the loop body is a single `+=` on a method call." **Fixed in session 248: use-after-free in `Array.prototype.pop`.** Root cause: `builtin_array_proto_pop` returned a `TVal` pointing at the popped HString, then `array_delete_elem` decref'd the array slot — and if the popped value's only remaining reference was that slot (i.e., the popped HString was interned and shared only by the slot we were removing), refcount hit 0 and the HString was removed from the string table and freed.
