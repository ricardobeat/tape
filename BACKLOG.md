## High Impact — RegExp (libregexp swap)

- [ ] **B32** — Unicode regex correctness: regex compilation now honors `\u{1F600}`, `\p{Letter}`, and `v`-flag string-sets, but `lre_exec` is invoked with `cbuf_type=0` (byte mode). That works for ASCII-only subjects and for property-escapes against a pure-ASCII input, but multi-byte UTF-8 subject strings break: `"\u{1F600}".match(/\u{1F600}/u)` returns `null` because libregexp interprets `cbuf_type=0` as "one byte per character" and never decodes the surrogate pair. Fix: convert UTF-8 subject bytes to UTF-16 in our wrapper and pass `cbuf_type=2` (matching QuickJS's wiring in `quickjs.c:47884`). When this lands the test262 `regexp-unicode-property-escapes` count (currently all-CE or fail) will start passing through, and `regexp-v-flag` closer to full conformance.

- [ ] **B33** — `d`-flag `match.indices` runtime: we now accept `d` at compile time (and call out `hasIndices=true` correctly), but `String.prototype.match` and `RegExp.prototype.exec` never populate `.indices` on the result array. Also `RegExp.prototype.hasIndices` accessor getter is not registered (the test262 skip list does not currently skip it, so it shows as fail rather than CE). Fix: extend the regex call sites to (a) extract capture offsets from the exec result and build an `indices` array of `[start,end]` pairs (and `indices.groups`), and (b) add `BUILTIN_REGEXP_PROTO_GET_HASINDICES` + `REGEXP_PROTO_GET_UNICODESETS` accessors mirroring the existing `dotAll`/`sticky` pattern in `src/builtins/regexp.c3` (~line 1265).

- [ ] **B34** — `String.prototype.match` with a non-global named-group regexp: `String.prototype.match(/(?<y>\d{4})/)` returns the plain array of the full match, but `result.groups` is left `undefined`. `RegExp.prototype.exec(/(?<y>\d{4})/.exec("2024-01"))` correctly returns `result.groups === {y:"2024"}`. Root cause in `src/builtins/string.c3` ~line 2110 (`builtin_string_proto_match` symbol-match fast path): after `re_exec` it sets the full match but never reads `re_get_groupnames(re)` to construct the `groups` object. Fix: mirror the same groupname iteration that `regexp_proto_exec` (`src/builtins/regexp.c3` ~line 354) already does, populating `result.groups[name] = capture_string` for each non-empty named capture. (Once B33 lands, populate `result.indices.groups` the same way.)

- [x] **B37** — Destructuring-binding parser holes closed across five surfaces. (a) `catch ([x, _]) {}` / `catch ({x}) {}` — catch-clause destructuring now parses and runs with block-scoped LET bindings and REQUIRE_OBJ for object patterns. (b) `for ([x, y] of arr) {}` / `for ({x} of arr) {}` — bare for-of/for-in destructuring heads now parse and run with VAR bindings. (c) `[a = function() { return b; }, b = 1]` — default-value ordering already correct; the remaining nested-object-default gap `[{x} = default]` in array patterns was fixed by consuming the closing `}` before the whole-object default and applying the default to the whole nested object value (with per-property defaults preserved). (d) `({ [computed()]: a } = obj)` — computed property keys in destructuring patterns now parse and execute. (e) `({ __proto__: x, ...y } = obj)` — object rest patterns in destructuring now parse and run. See commits `e6ca1d6`, `668cfe7`, `d3c3b3a`, `12c1642`, `7473dd3`, and the nested-object-default follow-up in this session.

- [x] **B38** — *(done — root cause was B43, now fixed; see B43 entry)* `generators` feature flag CEs. The bulk was missing generator-method shorthand in class bodies/object literals, resolved by the generators branch merge. Residual generator-adjacent CEs (computed-property-names from `yield`/`await` expressions) are tracked under B44's cluster, not a separate surface.
