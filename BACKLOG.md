Minimal repro: `test/test_bug_chained_and_comparison.js`. **Fixed in session 247: same fused-compare peephole as B26, second symptom. The `&&` operator emits `IF_FALSE left, +N` to short-circuit; the peephole fuses the preceding `GE/LE` with that IF_FALSE into a `JMP_Nxx`. The `JMP_Nxx` does NOT materialize the comparison result in `left` (the destination register the `&&` operator uses as its final result). On the short-circuit path, the later `LDREG left = right` is skipped — so `left` is left holding whatever the now-fused `GE` would have written… except the `GE` was removed by the fusion. Result: `left` is stale/uninitialized → returns `undefined`. Fix: in the peephole (`src/compiler/context.c3`), detect the `&&`/`||` skip pattern by checking whether the if_false's target is immediately preceded by an `LDREG` whose destination is `rA`; if so, refuse to fuse (leaving the original `CMP + IF_FALSE` in place preserves the materialised boolean). Verified by `test_bug_chained_and_comparison.js` (now passes) and the rosetta matrix_ops / caesar / l3 suites (still pass).**

- [x] **B28** — Two related array-mutation bugs, both rooted in the fused-compare peephole (`src/compiler/context.c3`) mis-handling `while (cond1 && cond2)` loops:
  - Classic three-statement swap chain (`var tmp = a[i]; a[i] = a[j]; a[j] = tmp;`) inside a `while` loop corrupts the array instead of swapping. Symptom: e.g. heapsort's siftDown produced `[1,5,null,3,2,9,8]` (length 7, not 6) for input `[5,3,8,1,9,2]`. Minimal repro: `test/test_bug_swap_chain_in_loop.js`.
  - `while (i + 1 < arr.length && arr[i + 1] === expected)` loop body that increments `i` and re-reads `arr[i + 1]` on the next iteration hangs indefinitely instead of terminating. Minimal repro (run under external `timeout`, no internal guard): `test/test_bug_nested_while_index_reread.js`.

  Found via the expanded rosetta suite (`test/rosetta/heap_sort.js`, `test/rosetta/shell_sort.js`, `test/rosetta/range_expansion.js`'s `compressRange`, and `test/rosetta/merge_sort.js`'s earlier version before it was reworked to avoid the pattern).

  **Fixed in session 248: same fused-compare peephole as B23/B26/B27, fourth symptom.** The peephole has a "bridge-pattern fix" (added for B23/B26) that detects when the if_false's destination is itself an if_false/if_true, and extends the fused offset by `bridge_sbx + 1` to skip the bridge. The fix was correctly handling the if/else-if/else case from B23/B26 — but it fired indiscriminately for any if_false/if_true at the destination, including the if_true *back edge* of a `while (cond1 && cond2)` loop. The structural difference: in the B23/B26 if-chain, the destination is a FORWARD continuation (`if_false` with positive offset, jumping to the next chain link). In a `while` loop with `&&`, the if_false short-circuit for cond1 lands on the LDREG that precedes the if_true back edge, and the if_true is a BACK edge with NEGATIVE offset. Extending `fused_off` by `bridge_sbx` (= negative offset) flipped the fused `JMP_Nxx` from a forward short-circuit to a backward loop-forever jump. Once the loop counter underflowed past the condition threshold, the fused JMP_Nxx kept firing and the loop spun (or, for heapsort, the bridge-skipping made the array appear to grow).

  Fix: only treat the if_false/if_true at bridge_slot as a bridge when its offset is positive (`bridge_sbx > 0`). A negative or zero offset means it's a loop back edge or self-targeting branch — leave `bridge_sbx = 0` and let the fused JMP_Nxx land on the bridge slot unchanged. This preserves the B23/B26 fix (forward continuations still skip correctly) while letting the while-loop back edge be the actual target.

  Verified by `test/rosetta/heap_sort.js` (now passes), `test/rosetta/shell_sort.js` (no longer hangs), `test/test_bug_nested_while_index_reread.js` (`compressRange` returns the right answer instead of timing out), and a new `test/test_b28_peephole_bridge.js` covering heapSort, shellSort, compressRange, countDown, findPairEnd (`&&` short-circuit), firstPositiveOrLong (`||` short-circuit), and untilTen (do-while). Rosetta 100/100. The peephole has now been the root cause of four distinct engine bugs (B23, B26, B27, B28); any new failure that involves `if/else if/else` chains or `&&`/`||` short-circuits in loops should start there.

- [ ] **B29** — `array_concat.js` (from `test/rosetta/FAILURES.md`): "`[1,2,3].concat([4,5,6])` engine crashes (RC 133)". **Re-investigated in session 248: not reproducible.** Tested 13 concat patterns — basic, multi-arg, with-strings, with-primitives, deeply-nested, 2000-element large concat, single-arg concat, deeply-nested arrays as elements, with-`null`/`undefined` elements, self-concat (`a.concat(a)`), `concat()` with no args (returns copy), pre/post increment of source during concat — all pass. The bug must have been fixed by the B14 generic array-method refactor (session 218) which made `Array.prototype.concat` a proper builtin dispatching through the user-overridable `Symbol.species`-aware path. `test/rosetta/FAILURES.md` updated to mark the entry as closed.

- [x] **B30** — `stack.js` `while`-loop variant (from `test/rosetta/FAILURES.md`): "`while (!st.isEmpty()) out += st.pop()` returns a partial/garbled string when the loop body is a single `+=` on a method call." **Fixed in session 248: use-after-free in `Array.prototype.pop`.** Root cause: `builtin_array_proto_pop` returned a `TVal` pointing at the popped HString, then `array_delete_elem` decref'd the array slot — and if the popped value's only remaining reference was that slot (i.e., the popped HString was interned and shared only by the slot we were removing), refcount hit 0 and the HString was removed from the string table and freed. The freed memory was then immediately reused by the next allocation triggered by `out += popped` (concat), overwriting the HString's data so the popped char read back as `""` (and on the next iteration, the add tried to convert a Symbol-shaped tag back to a string, throwing `Cannot convert a Symbol value to a string`). The bug only triggered when the source string contained a duplicate character and the duplicate's last array-slot reference was the one being popped — e.g. `["a","b","a","c"]` only misbehaves on the 4th pop (the second "a"), not on "abc" where all elements are unique. Fix: `builtin_array_proto_pop` now `incref`s the popped element before the slot is cleared, transferring ownership to the caller's result register (which decrefs normally when consumed). Verified by `test/test_b30_while_pop.js` (19 assertions covering while, for, and direct `items.pop()` paths) and the rosetta suite (100/100). Note: the failure was *not* in the call/return machinery (it happened inside the inner lightfunc) nor in the compound-assign peephole — it was a refcount ownership bug in the builtin itself.

## High Impact — RegExp (libregexp swap)

- [x] **B31** — Replaced the original-vendor `libregexp.c` (from QuickJS) with quickjs-ng's actively-maintained version. The fork brings ES2018/2020/2022/2024 regex support that the vendored copy lacked: `\u{1F600}` brace codepoint escapes, `\p{Letter}` Unicode property escapes, duplicate-named groups across alternatives, full `v`-flag string-set notation, and the per-capture `d`-flag `match.indices` runtime (QuickJS adds indices on top of libregexp's recorded flag). Our wrapper (`libregexp/re_wrapper.{h,c}`) now passes the `RE_FLAG_UNICODE`/`STICKY`/`INDICES`/`UNICODE_SETS` bits through to `lre_compile` and rejects `u`+`v` mutually exclusive flag combinations per ES2024. VM/parser/RegExp-constructor paths (`src/builtins/regexp.c3`, `src/vm/vm_objects.c3`, `src/builtins/string.c3`, `src/re_bindings.c3`) parse the new flag letters in literals and construct args. Test262 skip-list: removed `regexp-unicode-property-escapes`, `regexp-v-flag`, `regexp-duplicate-named-groups` — all three are now attempted (780 new tests visible in Phase 8 alone). Test262 skip-list also dropped `regexp-modifiers` is preserved (Stage 3 `(?i:...)` — not standardized). Roetta still 100/100; Phase 8 passes went from 1066 (pre-swap) to 1223 (with newly attempted tests visible as fails — the runtime gaps below). Remaining gaps in the same area now tracked as B32–B33.

- [ ] **B32** — Unicode regex correctness: regex compilation now honors `\u{1F600}`, `\p{Letter}`, and `v`-flag string-sets, but `lre_exec` is invoked with `cbuf_type=0` (byte mode). That works for ASCII-only subjects and for property-escapes against a pure-ASCII input, but multi-byte UTF-8 subject strings break: `"\u{1F600}".match(/\u{1F600}/u)` returns `null` because libregexp interprets `cbuf_type=0` as "one byte per character" and never decodes the surrogate pair. Fix: convert UTF-8 subject bytes to UTF-16 in our wrapper and pass `cbuf_type=2` (matching QuickJS's wiring in `quickjs.c:47884`). When this lands the test262 `regexp-unicode-property-escapes` count (currently all-CE or fail) will start passing through, and `regexp-v-flag` closer to full conformance.

- [ ] **B33** — `d`-flag `match.indices` runtime: we now accept `d` at compile time (and call out `hasIndices=true` correctly), but `String.prototype.match` and `RegExp.prototype.exec` never populate `.indices` on the result array. Also `RegExp.prototype.hasIndices` accessor getter is not registered (the test262 skip list does not currently skip it, so it shows as fail rather than CE). - [ ] **B34** — `String.prototype.match` with a non-global named-group regexp: `String.prototype.match(/(?<y>\d{4})/)` returns the plain array of the full match, but `result.groups` is left `undefined`. `RegExp.prototype.exec(/(?<y>\d{4})/.exec("2024-01"))` correctly returns `result.groups === {y:"2024"}`. - [ ] **B40** — Huge-length array-like handling: `Array.prototype` methods on `{length: 2^53-ish}` either loop for the full timeout (46 TIMEOUTs) or balloon memory until killed (MEMKILL).
