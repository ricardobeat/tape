Session 130: Compiler pushback fix — expect/match respect pushback buffer (+492 test262 passes).
Fixed critical compiler bug where `expect()` and `match()` read directly from the lexer,
completely ignoring the pushback token buffer. This caused any parenthesized single-identifier
expression like `(x)` to fail compilation: the arrow-detection path would consume and push
back `)`, `x`, `(`, but `expect(RPAREN)` in the grouping path would skip the pushed-back `)`
and read from the raw lexer. Changed `expect()` to use `advance()` (which respects pushback)
and `match()` to use `check()` + `advance()`. Impact: Phase 2 +340 (532→872), Phase 0-1 +49
(461→510), Phase 5 +32, Phase 6 +23, Phase 7 +17. quick.sh: 186→219 (+33) passes.

Session 131: UNM -0 preservation + ToPrimitive TypeError in operators (+7 quick.sh, +5 phases 3/5/6).

Session 134: needs_restart guards in all binary operators (+25 test262, +4 quick.sh).
Fixed critical bug where SUB, MUL, DIV, MOD, EXP, BAND, BOR, BXOR, SHL, SHR,
USHR, LT, LE, GT, GE handlers did not check `needs_restart` after the first
`vm_check_to_number_throw`. When lhs valueOf threw an error that was caught by

Session 135: Date ToPrimitive hint — toString before valueOf (ES5 §8.12.8).
Fixed spec compliance bug where `to_primitive_value()` always called valueOf()
before toString() for ALL objects. Per ES5 §8.12.8, Date objects must use
"string" hint (toString first, then valueOf), while all other objects use
"number" hint (valueOf first, then toString). The `+` operator calls
ToPrimitive with no explicit hint, which defaults differently for Date vs
other objects. Before the fix, `new Date(0) + new Date(0)` returned `0`
(numeric addition of epoch values) instead of the concatenated toString
representations. Added Date class check in `to_primitive_value()` to swap
the call order for Date objects. quick.sh 260/23/56 — no regressions.
Both nanbox and nonanbox builds pass.

Session 136: catch variable PUTVAR in all scopes (+676 test262 passes).
Fixed critical compiler bug where the catch variable was only written to the
environment via PUTVAR in global scope (`if (self.is_global)`), not in function
scope. The CATCH opcode correctly stores the thrown value in a register, but
GETVAR/TYPEOFIDENT resolve catch variables via environment lookup — so without
PUTVAR, the environment binding for `e` remained `undefined` even though the
register held the thrown object. This caused `typeof e` to return "undefined"
and `e === undefined` to be true in catch blocks inside functions, while
`"" + e` would call `toString()` on the stale register value and show
`[object Object]`. Root cause: line 1859 in compiler.c3 had `if (self.is_global)`
guarding the PUTVAR emission. Fix: removed the guard, always emit PUTVAR after
CATCH. Impact: Phase 0-1 511→533 (+22), Phase 1 35→46 (+11), Phase 2 924→944
(+20), Phase 3 2067→2296 (+229), Phase 4 79→83 (+4), Phase 5 2541→2799 (+258),
Phase 6 1060→1141 (+81), Phase 7 177→182 (+5), Phase 8 561→607 (+46).
quick.sh 260→269 (+9). Both nanbox and nonanbox builds pass.

## Root Cause Analysis: Top Failure Categories

### 1. String `===` comparison using pointer identity (~fixed, but cascading effects)
**Impact**: Every test using `===` on non-interned strings (e.g., `str[0] === "h"`).
**Root cause**: The `SEQ` opcode compares strings by `get_heapptr() == get_heapptr()`
(pointer identity), which is correct only if all strings are properly interned. The
GETPROP handler for string primitive index access (`str[i]`) and string object index
access allocated new HString objects without looking up the string table first, producing
different pointers than the interned copies of the same content.
**Status**: ✅ FIXED (Session 103) — Added `Heap.str_intern()` method that does
lookup-first. Replaced all `hstring_alloc` + `str_table_insert` call sites with
`str_intern()` in vm.c3, builtins.c3, and heap.c3. Also simplified `intern_string()`,
`vm_number_to_string()`, and `vm_fastint_to_string()` to use the new helper.

### 2. Function `.name` not set on JS functions (~2000+ tests)
**Impact**: Every test checking `fn.name` or property descriptors on function objects.
**Status**: ✅ FIXED — CLOSURE opcode now creates `.name` property from template name.

### 3. Lightfunc constructors lack `.prototype` (~1500+ tests, cascading)
**Status**: ✅ FIXED (Session 101) — Added `.prototype` case in lightfunc GETPROP
handler with `lightfunc_get_proto()` helper mapping fn_index → heap proto fields.

### 4. Built-in method `.name` absent on lightfunc methods (~800+ tests)
**Status**: ✅ FIXED (Session 101) — `getOwnPropertyDescriptor` handler synthesizes
`.name` and `.length` descriptors for lightfuncs.

### 5. Property descriptor flags wrong on built-ins (~600+ tests)
**Status**: ✅ FIXED (Session 102) — Changed all method registrations to `PROP_FLAGS_NWC`.

### 6. String wrapper indexed properties missing (~70+ tests)
**Impact**: `Object.keys(new String("abc"))`, `for-in`, `getOwnPropertyDescriptor`,
`hasOwnProperty` all returned empty results for String wrapper objects.
**Root cause**: The String constructor only set `.length` as an own property.
ES5 §15.5.5.1 requires indexed character properties ("0", "1", ...) as own
non-writable, enumerable, non-configurable properties.
**Status**: ✅ FIXED (Session 104) — Added `populate_string_indexed_props()` helper
that eagerly creates indexed properties. Applied to all 5 String wrapper creation
sites: `builtin_string()`, `builtin_to_object()`, `builtin_object()`, INITFOR,
and with-statement. Also restructured `getOwnPropertyDescriptor` to run the
string exotic fallback AFTER the default lookup (per spec). Added `PROP_FLAGS_NEC`
constant for non-writable/enumerable/non-configurable.
